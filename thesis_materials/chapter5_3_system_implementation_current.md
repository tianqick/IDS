# 5.3 系统实现

在完成系统总体设计、模型设计、功能模块设计和数据库设计之后，本文进一步基于当前项目环境完成了网络入侵检测系统的工程实现。系统整体采用 B/S 架构和前后端分离思想，前端负责页面交互、数据展示和图表渲染，后端负责用户认证、权限控制、任务调度、模型推理、结果归档、告警处理和在线流量监测等业务逻辑。系统核心技术包括 Flask、Vue 3、MySQL、SQLAlchemy、PyTorch、Scapy 和 Python cicflowmeter 等。

从实现结果来看，系统已经形成了较为完整的检测业务闭环。普通用户可以完成登录、数据上传、检测任务提交、任务进度查看、检测结果查询和历史记录查看；管理员在此基础上还可以进行告警处理、模型维护、用户管理和在线流量监测。系统将离线 CSV 检测与在线流量监测两类场景进行统一管理，使训练完成的深度学习模型能够在 Web 平台中完成实际推理应用。

## 5.3.1 前后端整体实现

系统前端采用 Vue 3 构建单页应用，页面入口由后端统一渲染基础 HTML 模板，随后由前端脚本完成页面状态管理、菜单切换、表单提交和结果渲染。前端页面主要包括登录界面、系统首页、上传检测、检测结果、告警中心、模型管理、用户管理和在线流量监测等视图。页面布局采用侧边栏导航方式，用户登录后可以根据自身角色看到不同的功能入口。

后端采用 Flask 作为 Web 服务框架，并通过 Blueprint 对前端页面和业务接口进行组织。系统启动后，前端所有业务请求均由后端统一处理，后端根据用户会话判断登录状态，并根据角色执行权限校验。普通用户只能访问与本人相关的检测任务和检测记录；管理员可以访问系统级数据，并具备模型维护、告警处理、用户管理和流量监测等管理权限。

在数据访问方面，系统使用 SQLAlchemy ORM 对数据库表进行映射，将用户、模型、数据集、检测任务、检测记录、攻击结果、告警日志和操作日志等业务对象封装为模型类，降低了直接编写 SQL 的复杂度。文件类资源则保存在服务器本地目录中，数据库中保存文件路径和元数据，从而实现结构化业务数据与非结构化文件资源的分离。

## 5.3.2 用户登录与权限控制实现

用户登录是系统的入口功能。用户在登录界面输入用户名和密码后，前端将登录信息提交给后端进行身份校验。后端首先根据用户名查询用户信息，再校验密码是否正确；验证通过后，将用户编号、用户名和角色写入会话中，并记录登录操作日志。对于早期未加密的演示账号密码，系统在登录时会自动完成哈希化处理，保证后续密码存储更加安全。

权限控制主要体现在前端菜单控制和后端访问控制两个方面。前端根据当前用户角色动态生成导航菜单，普通用户只显示系统首页、上传检测和检测结果等基础功能；管理员则额外显示告警中心、在线流量监测、模型管理和用户管理等功能。后端在涉及管理操作的业务处理前进行角色校验，防止普通用户越权访问管理员功能。

在数据访问范围上，系统会根据用户角色对查询结果进行过滤。管理员可以查看全平台的检测记录和统计数据，普通用户只能查看自己提交的检测任务和检测结果。通过会话状态、角色判断和数据范围过滤的组合，系统实现了较完整的登录认证和权限隔离。

## 5.3.3 系统首页与统计展示实现

系统首页用于展示平台运行概况。用户登录后，前端自动请求首页统计数据，后端根据当前用户角色计算可见范围内的统计结果。统计内容主要包括检测记录数量、数据集数量、告警数量、累计检测样本数、累计攻击样本数、累计正常样本数、攻击类型分布、近期攻击趋势、最近检测记录和最近告警信息等。

后端在统计过程中对检测记录、攻击结果和告警日志进行聚合查询，并根据角色过滤数据范围。普通用户首页展示个人相关检测统计，管理员首页展示全局统计。前端接收到统计数据后，通过统计卡片、饼图、折线图和列表组件进行可视化展示，使用户能够快速了解系统当前运行状态和历史检测趋势。

系统首页还实现了服务器访问地址展示和二维码辅助访问功能。当系统检测到可访问地址时，前端可以生成二维码，便于同一局域网内其他终端快速访问系统。该功能增强了系统演示和调试过程中的便利性。

## 5.3.4 数据上传与后台检测任务实现

数据上传与后台检测任务是系统的核心业务功能。用户进入上传检测页面后，首先选择当前使用的检测模型，然后上传待检测的 CSV 文件。系统在后端对文件类型进行校验，仅允许符合要求的 CSV 文件进入检测流程。文件保存时，系统会使用安全文件名和时间戳生成实际存储名称，避免同名文件覆盖，并将文件名、保存路径、样本数量和上传时间写入数据集信息表。

由于流量数据文件可能包含大量样本，若直接在请求过程中完成模型推理，容易造成页面等待时间过长。为此，系统采用后台任务方式执行检测。文件保存完成后，后端会创建一条检测任务记录，将任务状态初始化为排队中，并立即向前端返回任务信息。随后系统启动后台线程执行检测流程，前端则通过定时刷新任务状态的方式展示处理进度。

任务执行过程中，后台线程会将任务状态更新为执行中，并在处理数据块时持续更新已处理样本数。若检测成功，系统将任务状态更新为已完成，并关联最终生成的检测记录；若检测过程中出现异常，则将任务状态更新为失败，并保存失败原因。通过该实现方式，系统既保证了大文件检测任务能够持续执行，也避免了前端页面长时间阻塞。

## 5.3.5 模型推理与结果写入实现

后台检测任务进入模型推理阶段后，系统会加载当前选择的深度学习模型及其预处理参数。模型服务首先读取待检测 CSV 文件，并按块进行处理，以降低一次性读取大文件带来的内存压力。每个数据块会经过字段清洗、非特征字段剔除、缺失值填充、标准化和张量重构等步骤，最终转换为模型可接受的输入格式。

在模型加载方面，系统能够根据模型权重文件推断模型结构信息，包括是否包含卷积层、循环层、双向结构和注意力层，以及输出类别数等。这样可以提高模型服务的适配能力，避免模型信息与权重文件不一致导致加载失败。模型完成推理后，系统通过 Softmax 得到预测概率，并将概率最高的类别作为当前样本的检测结果，同时保存对应置信度。

检测结果写入数据库时，系统会分别统计样本总数、正常流量数量和攻击流量数量。对于被判定为攻击的样本，系统保存攻击类型、风险等级、置信度、源地址和目的地址等信息，并同步生成告警日志。为提高数据库写入效率，攻击结果和告警信息采用缓冲批量写入方式，当缓冲数量达到设定阈值后统一提交。检测完成后，系统更新检测记录中的统计字段，并写入操作日志。

## 5.3.6 检测结果与历史记录实现

检测结果页面用于展示系统已完成的检测记录。前端加载结果列表时，后端会根据当前登录用户角色返回对应范围内的数据。管理员可以查看全部检测记录，普通用户只能查看本人提交的记录。列表中主要展示源文件名、检测模型、样本总数、正常流量数量、攻击流量数量、检测时间、操作者和记录状态等内容。

用户点击某条检测记录后，系统进入检测详情页面。后端根据检测记录编号查询对应的攻击结果明细，并返回攻击类型、风险等级、置信度、源 IP、目的 IP 和创建时间等信息。前端将这些信息展示为明细表格，帮助用户进一步分析本次检测中识别到的异常流量。

对于管理员，系统还提供检测记录删除能力。删除记录时，后端会同步清理关联的攻击结果和告警信息，并解除检测任务与检测记录之间的关联，保证数据库中不会残留无效关联数据。该功能便于管理员清理测试数据或无效检测记录。

## 5.3.7 告警中心实现

告警中心用于集中管理检测过程中生成的异常流量告警。系统在识别到攻击流量后，会自动生成告警记录，告警内容包括攻击类型、风险等级、关联检测记录和创建时间等信息。管理员进入告警中心后，可以按照告警处理状态进行筛选，并通过分页方式查看告警列表。

告警处理状态主要包括未处理、已处理和忽略。管理员根据实际分析结果更新告警状态后，后端会将状态变更保存到数据库，并记录对应操作日志。该模块将模型输出的攻击识别结果进一步转化为可管理的安全事件，使系统不仅能够完成检测，还能支持后续告警处置和运维追踪。

当前项目的告警中心重点面向在线流量监测场景展示告警信息，因此在页面中更适合结合在线监测结果进行截图说明。对于离线检测任务中产生的攻击结果，用户可在检测详情页面中查看对应明细。

## 5.3.8 模型管理与模型能力展示实现

模型管理模块面向管理员开放，用于维护系统可用的深度学习模型。管理员可以新增模型记录，填写模型名称、模型路径、模型类型、准确率、精确率、召回率、F1 值、误报率、漏报率、推理时延、数据格式说明和所需字段列表等信息。若上传新的模型文件，系统会校验文件后缀是否符合要求，并将模型文件保存到指定目录。

在保存模型信息前，系统会检查模型名称、模型路径和数据格式说明是否完整，同时校验模型文件是否存在，避免无效模型进入检测链路。系统同一时刻只允许一个模型处于启用状态，当管理员激活某个模型时，系统会自动取消其他模型的启用状态，保证后续检测任务始终使用明确的模型版本。

除模型管理外，系统还实现了模型能力展示面板。用户可以查看当前模型的 Accuracy、Precision、Recall、F1-Score、FPR、FNR、推理时延、历史检测样本数、异常样本数、正常样本数和输入字段要求等信息。该功能将模型实验指标和系统业务数据结合展示，便于用户了解模型能力和数据格式要求。

## 5.3.9 用户管理与操作日志实现

用户管理模块仅对管理员开放，主要用于维护系统账号和角色信息。管理员可以新增用户、修改用户信息、修改用户角色和删除无关联记录的用户。新增用户时，系统会校验用户名和密码是否为空、角色是否合法以及用户名是否重复；修改用户时，系统同样会检查用户名唯一性和角色合法性，并在需要修改密码时重新生成哈希密码。

为了保证数据安全和历史记录可追溯，系统限制管理员删除当前登录账号；如果目标用户已经存在关联检测记录，系统也不会直接删除该用户，以避免历史检测记录失去所属用户信息。用户管理中的新增、修改和删除操作都会写入操作日志，便于后续审计。

操作日志贯穿系统多个关键模块，包括登录、退出登录、创建检测任务、完成检测任务、模型新增、模型启用、模型删除、告警状态更新、用户新增、用户修改和用户删除等操作。通过操作日志，管理员可以在后续排查问题时了解关键业务行为的发生时间和操作者。

## 5.3.10 在线流量监测实现

在线流量监测模块是当前系统从离线文件检测扩展到在线巡检场景的重要功能。该模块面向管理员开放，主要实现抓包网卡配置、监测启动与停止、运行状态查看、PCAP 测试提取、流量特征生成和自动检测归档等功能。

在线监测启动前，管理员需要选择或配置抓包网卡。后端会检查 Scapy 和 Python cicflowmeter 是否可用，并根据网卡配置判断监测链路是否具备运行条件。监测启动后，系统创建后台监测线程，周期性执行抓包、特征提取和自动检测流程。抓包阶段使用 Scapy 从指定网卡采集网络数据并生成 PCAP 文件；特征提取阶段使用 Python cicflowmeter 将 PCAP 转换为流量特征 CSV；字段规范化阶段将提取结果转换为模型可接受的 CIC-IDS 风格字段；最后系统调用当前启用模型完成自动检测。

监测过程中，系统会记录最近抓包时间、最近提取时间、最近处理文件、生成的 PCAP 文件名、生成的 CSV 文件名、处理次数、空特征文件次数和异常信息等状态。若提取到的 CSV 中没有有效数据行，系统会将文件归档并更新空特征计数；若存在有效流量样本，则调用检测流程生成检测记录、攻击结果和告警信息。通过该模块，系统实现了从“原始网络流量采集”到“模型检测”和“告警生成”的自动化处理链路。

## 5.3.11 本节小结

本节围绕当前项目的系统实现过程进行了说明。系统前端采用 Vue 3 实现单页应用和可视化交互，后端采用 Flask 完成业务接口、权限控制和任务调度，数据库采用 MySQL 保存结构化业务数据，模型推理层通过 PyTorch 完成深度学习模型加载与批量预测，在线流量监测层则结合 Scapy 和 Python cicflowmeter 完成抓包与特征提取。通过上述实现，系统已经具备离线 CSV 检测、检测结果展示、告警管理、模型管理、用户管理和在线流量监测等功能，能够支撑基于深度学习的网络入侵检测应用场景。
