# 4.5 数据库设计

数据库设计是系统设计中的重要组成部分，其合理性直接影响系统数据存储、业务查询、结果追溯和后续维护的效率。本文所设计的网络入侵检测系统采用 MySQL 作为关系型数据库，用于保存用户账号、模型信息、数据集信息、检测任务、检测记录、攻击结果、告警日志和操作日志等业务数据。与此同时，模型权重文件、上传数据文件、在线抓包文件和特征文件等体积较大的文件资源保存在服务器本地目录中，数据库中仅记录其路径和相关元数据，从而避免数据库承担过重的文件存储压力。

结合系统功能模块划分，数据库设计需要满足以下要求：第一，能够支持管理员与普通用户两类角色的数据隔离和权限管理；第二，能够记录离线检测和在线流量监测过程中产生的任务状态、检测结果和告警信息；第三，能够保存模型元数据和评价指标，支撑模型启用、切换和能力展示；第四，能够通过检测记录、攻击结果和告警日志之间的关联关系，实现检测结果的追踪和复查；第五，能够记录关键操作行为，为系统审计和问题定位提供依据。

## 4.5.1 概念结构设计

概念结构设计主要用于从系统业务需求中抽象出核心实体及其相互关系。根据本文系统的功能需求，数据库中的主要实体包括系统用户、模型信息、数据集信息、检测任务、检测记录、攻击结果、告警日志和操作日志。

系统用户实体用于保存登录账号、密码和角色信息，是系统权限控制的基础。模型信息实体用于保存深度学习模型的名称、路径、类型、评价指标、输入格式说明和启用状态，支撑系统中的模型管理和模型能力展示功能。数据集信息实体用于记录用户上传的数据文件，包括文件名称、保存路径、样本数量和上传时间。检测任务实体用于描述后台异步检测任务的执行过程，包括任务发起用户、源文件、任务状态、总样本数和已处理样本数等信息。检测记录实体用于保存一次检测任务完成后的总体统计结果，包括样本总数、正常流量数量、攻击流量数量和使用模型信息。攻击结果实体用于保存被模型识别为异常流量的样本信息。告警日志实体用于保存由攻击结果触发的告警内容、告警级别和处理状态。操作日志实体用于记录用户登录、模型切换、记录删除和用户管理等关键操作。

从实体关系来看，一个系统用户可以发起多条检测任务，也可以拥有多条检测记录；一条检测任务在执行完成后可以关联一条检测记录；一条检测记录可以对应多条攻击结果和多条告警日志；一个模型可以被多条检测任务或检测记录引用；操作日志通过操作者名称与用户行为形成逻辑关联。通过上述实体和关系设计，系统能够形成从用户提交任务、后台执行检测、保存检测结果、生成告警到后续审计追踪的完整数据链路。

## 4.5.2 逻辑结构设计

逻辑结构设计是在概念结构设计的基础上，将实体及实体关系转换为关系型数据库能够存储和管理的数据表结构。本文系统数据库采用主键、外键、唯一约束和逻辑关联字段共同维护数据关系。其中，用户表、检测记录表、攻击结果表、告警日志表和检测任务表之间设置了明确的外键关系；模型信息与检测任务、检测记录之间通过模型编号形成逻辑关联；操作日志主要保存操作行为本身，通过用户名与系统用户形成逻辑关联。

系统主要数据表及其功能说明如表4.5-1所示。

表4.5-1 主要数据表逻辑结构说明

| 表名 | 主键 | 关键外键或关联字段 | 主要功能 |
| --- | --- | --- | --- |
| user | id | username 唯一约束 | 保存系统用户账号、密码和角色信息 |
| model_info | id | is_active | 保存模型路径、评价指标、输入格式和启用状态 |
| dataset_info | id | file_path | 保存上传数据文件的来源、路径和样本数量 |
| detection_task | id | user_id、record_id、model_id | 保存后台检测任务状态、进度和文件路径 |
| detect_record | id | user_id、model_id | 保存每次检测任务完成后的总体统计结果 |
| attack_result | id | record_id | 保存攻击样本的类型、风险等级、置信度和地址信息 |
| alarm_log | id | record_id | 保存异常流量触发的告警内容和处理状态 |
| operation_log | id | username | 保存用户关键操作行为和操作时间 |

在表关系设计方面，user 表与 detection_task 表之间为一对多关系，一个用户可以发起多条检测任务；user 表与 detect_record 表之间也为一对多关系，一个用户可以拥有多条检测记录。detect_record 表与 attack_result 表之间为一对多关系，一次检测记录可以包含多条攻击样本明细；detect_record 表与 alarm_log 表之间为一对多关系，一次检测记录可以触发多条告警。detection_task 表通过 record_id 与 detect_record 表建立任务执行结果关联，当后台任务完成后，系统将检测结果写入 detect_record 表，并回填任务记录中的关联编号。

model_info 表主要用于保存模型元数据。由于检测记录需要保留任务执行时所使用模型的名称和路径快照，因此 detect_record 表中除了保存 model_id 外，还保存 model_name 和 model_path 字段。这样即使后续管理员修改模型信息，历史检测记录仍能保留当时的模型使用情况，提高结果追溯的可靠性。dataset_info 表主要记录上传文件信息，与检测任务之间通过文件路径和业务流程形成逻辑联系。operation_log 表用于记录操作审计信息，为避免用户删除或账号调整影响日志留存，日志表保存用户名字段作为逻辑关联信息。

## 4.5.3 数据字典设计

数据字典用于对数据库中各数据表的字段含义、数据类型和约束关系进行详细说明。通过数据字典设计，可以使数据库结构更加清晰，便于后续系统维护、功能扩展和论文说明。本文系统主要数据表结构如下。

### （1）系统用户表 user

user 表用于保存系统登录用户的基础信息，包括用户名、密码和角色。系统通过 role 字段区分管理员和普通用户，并根据角色控制前端菜单和后端业务权限。该表结构如表4.5-2所示。

表4.5-2 user 用户信息表

| 字段名 | 类型 | 键 | 允许空 | 说明 |
| --- | --- | --- | --- | --- |
| id | INT | PK | 否 | 用户编号 |
| username | VARCHAR(50) | UNIQUE | 否 | 登录用户名 |
| password | VARCHAR(255) | - | 否 | 登录密码，系统创建和修改时进行哈希处理 |
| role | VARCHAR(20) | - | 否 | 用户角色，主要包括 admin 和 user |
| create_time | DATETIME | - | 是 | 用户创建时间 |

### （2）模型信息表 model_info

model_info 表用于保存系统中接入的深度学习模型信息，包括模型路径、模型类型、评价指标、数据格式说明和启用状态。该表既服务于管理员模型管理，也服务于普通用户查看模型能力说明。表结构如表4.5-3所示。

表4.5-3 model_info 模型信息表

| --- | --- | --- | --- | --- |
| id | INT | PK | 否 | 模型编号 |
| model_name | VARCHAR(100) | - | 否 | 模型名称 |
| model_path | VARCHAR(255) | - | 否 | 模型权重文件路径 |
| model_type | VARCHAR(50) | - | 是 | 模型类型 |
| accuracy | FLOAT | - | 是 | 准确率 |
| metric_precision | FLOAT | - | 是 | 精确率 |
| metric_recall | FLOAT | - | 是 | 召回率 |
| metric_f1_score | FLOAT | - | 是 | F1 值 |
| metric_fpr | FLOAT | - | 是 | 误报率 |
| metric_fnr | FLOAT | - | 是 | 漏报率 |
| metric_inference_latency_ms | FLOAT | - | 是 | 推理时延 |
| description | TEXT | - | 是 | 模型说明 |

| create_time | DATETIME | - | 是 | 记录创建时间 |

### （3）数据集信息表 dataset_info

dataset_info 表用于记录用户上传的数据文件信息，主要保存文件名称、文件保存路径、样本数量和上传时间。该表为数据来源追踪和历史文件管理提供基础信息。表结构如表4.5-4所示。

表4.5-4 dataset_info 数据集信息表

| dataset_name | VARCHAR(100) | - | 否 | 原始数据文件名称 |
| file_path | VARCHAR(255) | - | 否 | 文件保存路径 |
| sample_count | INT | - | 是 | 样本数量 |
| upload_time | DATETIME | - | 是 | 上传时间 |

### （4）检测任务表 detection_task

detection_task 表用于保存后台异步检测任务的执行状态。用户提交检测文件后，系统会先创建检测任务记录，再由后台线程执行具体检测流程。前端通过任务状态和进度字段展示检测进度。表结构如表4.5-5所示。

| processed_rows | INT | - | 是 | 已处理样本行数 |
| status | VARCHAR(20) | - | 否 | 任务状态 |
| message | VARCHAR(255) | - | 是 | 任务提示信息 |
| create_time | DATETIME | - | 是 | 任务创建时间 |
| update_time | DATETIME | - | 是 | 任务更新时间 |

### （5）检测记录表 detect_record

detect_record 表用于保存一次检测任务完成后的总体结果，包括检测文件、使用模型、样本总数、正常流量数量和攻击流量数量等信息。该表是检测结果展示和历史记录查询的核心数据表。表结构如表4.5-6所示。

attack_result 表用于保存检测过程中被识别为异常流量的样本结果。系统会记录攻击类型、风险等级、预测置信度、源地址和目的地址等信息，便于用户查看具体攻击明细。表结构如表4.5-7所示。

alarm_log 表用于保存攻击流量触发的告警信息。每条告警与检测记录关联，管理员可以根据告警级别和处理状态对异常事件进行跟踪和处置。表结构如表4.5-8所示。

## 4.5.4 数据库设计小结

本文数据库围绕入侵检测系统的核心业务流程进行设计，通过 user、model_info、dataset_info、detection_task、detect_record、attack_result、alarm_log 和 operation_log 等数据表，分别支撑用户权限管理、模型管理、数据上传、后台检测、结果展示、告警处置和操作审计等功能。整体结构既能够满足系统当前的检测业务需求，也为后续增加新的模型版本、扩展检测任务类型和完善审计功能提供了数据层基础。
