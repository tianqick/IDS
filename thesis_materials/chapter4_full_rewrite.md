# 第4章 基于深度学习的入侵检测系统设计

## 4.1 系统总体设计

### 4.1.1 设计目标

本文所设计的基于深度学习的入侵检测系统，不仅需要完成网络流量数据的分类识别任务，还应具备较好的系统可用性、扩展性和管理能力。因此，在系统总体设计过程中，主要围绕以下几个目标展开。

首先，要保证系统具备较高的检测准确性。入侵检测系统的核心任务是对正常流量和攻击流量进行区分，并尽可能准确地识别具体攻击类型。因此，系统在模型层面采用深度学习方法，通过融合一维卷积神经网络、双向长短期记忆网络和注意力机制，提高模型对复杂流量特征的表达能力。

其次，要保证系统具备较好的处理效率。考虑到上传数据检测和在线流量监测场景中可能涉及较大规模数据处理任务，系统采用后台异步检测机制，以减少前端阻塞现象，提高用户交互效率。同时，在模型推理环节尽量减少重复预处理和无效计算，从而提升整体检测速度。

再次，要保证系统具有良好的可用性。系统面向管理员和普通用户两类角色设计，用户可以通过浏览器完成登录、检测、结果查看和管理操作，界面布局应尽可能清晰直观，功能操作应尽可能简洁明确，以降低使用门槛。

最后，要保证系统具备一定的可扩展性。随着后续研究的深入，系统可能需要接入新的数据集、新的检测模型或新的流量处理模块。因此，在总体架构设计上应尽量保持模块之间的低耦合关系，使系统在后续功能扩展和维护时具有较好的适应能力。

### 4.1.2 系统整体架构设计

本文所设计的入侵检测系统采用 B/S 架构，并结合前后端分离的设计思想进行实现。系统整体上可以分为用户层、表示层、业务逻辑层和数据与支撑层四个部分。系统整体架构如图4.1所示。

用户层主要包括管理员和普通用户两类角色。普通用户负责执行检测任务、查看结果和查询历史记录；管理员在此基础上还承担模型维护、用户管理、告警处理、性能评估和流量监测等职责。

表示层主要由 Vue 前端界面组成，负责实现系统首页、上传检测、结果展示、模型管理、用户管理和流量监测等页面功能。前端通过浏览器与后端进行交互，并将检测结果、统计指标和任务状态以图表、卡片和列表等方式展示给用户。

业务逻辑层由 Flask API 接口层及各类服务模块构成，负责处理登录认证、权限校验、任务调度、数据查询、结果归档、模型维护和监测控制等核心逻辑。该层既是前端请求的处理中心，也是模型服务和数据库之间的桥梁。

数据与支撑层主要由模型服务层、MySQL 数据库、文件存储区和在线流量处理链路构成。模型服务层完成预处理参数加载、数据标准化和 PyTorch 模型推理；MySQL 数据库存储用户、模型、检测记录、攻击结果、告警日志和任务状态等数据；文件存储区保存上传的 CSV 文件、模型权重文件、PCAP 文件、归档特征文件以及评估元数据；在线流量处理链路则由 Scapy 抓包、Python cicflowmeter 特征提取和列名规范化送检三部分构成。

通过上述架构设计，系统形成了从前端交互、后台业务处理、模型推理到数据归档的完整工作链路，同时能够兼顾离线 CSV 检测与在线流量监测两种应用模式。

## 4.2 数据预处理设计

### 4.2.1 数据集选取与分析

本文选用 CIC-IDS2017 数据集作为入侵检测模型训练与测试的数据基础。该数据集在入侵检测研究中应用广泛，包含正常流量以及多种典型攻击流量，能够较好地反映复杂网络环境下的流量分布特征。系统实验中涉及的流量类型包括 BENIGN、Bot、DDoS、DoS GoldenEye、DoS Hulk、DoS Slowhttptest、DoS slowloris、FTP-Patator、Heartbleed、Infiltration、PortScan、SSH-Patator 以及多类 Web Attack，共形成 15 类分类目标。

从数据特点来看，原始数据既包含流标识字段，也包含可用于分类的统计性特征字段。由于深度学习模型更关注稳定的数值型统计特征，因此在进入训练和检测阶段之前，需要对原始数据进行统一清洗和结构化处理。

### 4.2.2 数据清洗与缺失值处理

在原始网络流量数据中，不同样本可能存在空值、无穷值和异常格式等情况。为保证模型训练和检测稳定性，本文首先将正无穷和负无穷值替换为空值，再对数值型字段中的空值采用列均值进行填充。该方法在保留样本规模的同时，能够减轻异常值对模型训练带来的干扰。

同时，系统在训练阶段删除了 `Flow ID`、`Source IP`、`Source Port`、`Destination IP`、`Destination Port` 和 `Timestamp` 等标识性字段。这些字段更适合用于结果解释和追踪，而不是直接作为泛化分类特征输入模型。

### 4.2.3 特征选择与标准化

在完成数据清洗后，系统最终保留 78 个有效流量统计特征作为模型输入。这些特征主要涵盖流持续时间、数据包数量、包长度统计、流速率、包间隔、标志位计数和活动时间等内容，能够从多个角度描述网络连接行为。

由于不同特征的量纲差异较大，若直接输入模型可能导致训练不稳定。因此，本文采用 Z-Score 标准化方法对特征进行归一化处理，使各维特征在统一尺度下参与模型训练与推理。标准化参数在预处理阶段统一保存到 `preprocess_meta.json` 文件中，以保证训练和推理过程使用一致的均值和标准差。

### 4.2.4 数据集划分

为科学评估模型性能，本文对样本数据采用训练集、验证集和测试集划分策略。具体实现中，先从全部样本中抽取 10% 作为测试集，再从剩余样本中划分出约 10% 作为验证集，最终形成约 80% 训练集、10% 验证集和 10% 测试集的结构。为了保证实验可复现性，数据划分过程中设置固定随机种子。

在进入模型之前，系统还需将二维特征矩阵重构为三维张量输入格式，即将每个样本表示为“特征维度 × 1”的形式，以便后续一维卷积层和双向长短期记忆网络对其进行处理。

## 4.3 入侵检测模型设计

### 4.3.1 模型总体结构

结合网络流量数据既包含局部统计特征又具有一定序列相关性的特点，本文设计了一种融合一维卷积神经网络、双向长短期记忆网络和注意力机制的混合模型。模型整体由输入层、一维卷积特征提取层、BiLSTM 时序建模层、注意力加权层和全连接分类层组成。

输入层接收经过标准化后的 78×1 特征序列；一维卷积层负责提取局部模式特征；BiLSTM 层负责挖掘前后文依赖关系；注意力层用于突出对分类更关键的特征位置；最后通过全连接层和 Softmax 输出各类别的预测概率。

### 4.3.2 1D-CNN 特征提取层设计

一维卷积层采用输入通道为 1、输出通道为 64、卷积核大小为 3 的结构，并在卷积后接批归一化层和 ReLU 激活函数。随后通过最大池化层完成特征压缩。该设计能够较好地提取网络流量特征间的局部关联模式，为后续时序建模提供更加抽象的输入表示。

### 4.3.3 BiLSTM 时序建模层设计

卷积层输出的序列特征进一步送入两层 BiLSTM 网络进行双向时序建模。BiLSTM 通过前向和后向两个方向同时学习序列依赖关系，能够综合利用特征前后文信息，从而提高模型对复杂攻击模式的刻画能力。

### 4.3.4 注意力机制层设计

在 BiLSTM 输出之后，本文引入注意力机制，对不同时间步的隐藏状态赋予不同权重。这样可以增强模型对关键特征位置的关注程度，减轻固定长度向量表示带来的信息损失，提高复杂攻击行为识别效果。

### 4.3.5 输出层与损失函数设计

输出层采用两层全连接结构完成分类，其中中间层引入 Dropout 机制以减少过拟合风险，最终输出层通过 Softmax 生成 15 类预测概率。训练过程中采用交叉熵损失函数，优化器使用 Adam，学习率设置为 0.001，从而在保证训练稳定性的同时提高收敛效率。

## 4.4 系统功能模块设计

结合系统前端页面结构和后台接口实现，本文将系统功能划分为用户登录与权限控制、系统首页与统计展示、数据上传与检测任务、检测结果展示、历史记录管理、告警管理、模型管理、性能评估、流量监测和用户管理等十个功能模块。各模块共同构成系统完整的业务流程。

### 4.4.1 用户登录与权限控制模块设计

用户登录与权限控制模块是系统的入口模块，负责完成用户身份认证、角色识别与访问权限控制。用户在登录页输入用户名和密码后，后端通过认证逻辑校验登录信息的合法性；验证通过后，系统进一步查询当前用户所属角色，并将角色信息写入会话中。随后，前端根据角色动态加载对应的系统菜单和功能页面资源。

本系统主要区分管理员和普通用户两类角色。普通用户仅能执行上传检测、结果查看和历史记录查询等基础功能；管理员则拥有模型管理、用户管理、告警管理、性能评估和流量监测等更高权限。通过该模块，系统实现了功能隔离和权限边界控制，保证不同角色用户只能访问其被授权的功能区域。模块流程如图4.2所示。

图4.2：`fig4-2-login-permission-flow.png`

### 4.4.2 系统首页与统计展示模块设计

系统首页与统计展示模块用于展示平台整体运行概况，是用户进入系统后的首个功能界面。当前端加载首页时，后端会对检测记录数、数据集数量、告警总数、攻击类型分布和近期攻击趋势等信息进行统计计算，并将结果以 JSON 形式返回前端。前端再将这些数据渲染为统计卡片、折线图、分布图和最近记录列表。

需要说明的是，管理员与普通用户在首页可见的数据范围并不完全相同。普通用户仅能查看与自身账号相关的检测记录、结果统计和历史趋势；管理员则可以查看整个系统的全局数据。该模块通过对统计结果进行角色过滤，使首页既能反映平台总体运行状况，也能满足不同角色的个性化浏览需求。模块流程如图4.3所示。

### 4.4.3 数据上传与检测任务模块设计

数据上传与检测任务模块是普通用户和管理员最常使用的核心业务模块。用户在该模块中首先选择当前使用的检测模型，再上传待检测的 CSV 文件。系统接收到文件后，会创建一条后台检测任务记录，并同步记录任务状态、总行数和已处理行数等信息。检测工作在后台线程中执行，从而避免大文件处理时前端界面阻塞。

在任务执行过程中，系统按块读取数据文件，对每个数据块进行预处理、标准化和模型推理。检测结束后，系统生成检测记录、攻击结果和告警信息，并通知前端刷新任务状态。前端会通过定时轮询任务接口的方式显示进度条，并在任务完成后自动跳转到检测结果详情页面。模块流程如图4.4所示。

### 4.4.4 检测结果展示模块设计

检测结果展示模块主要负责将一次检测任务的总体结果和详细攻击信息直观呈现给用户。当前端请求检测结果列表或某条记录详情时，后端会联合查询 `detect_record`、`attack_result` 和 `alarm_log` 等表，并返回样本总数、正常流量数量、攻击流量数量、攻击类型、风险等级、源地址、目的地址和置信度等信息。

对于普通用户，该模块主要用于查看任务结果和详细分析；对于管理员，则还允许执行删除检测记录等管理操作，并在删除时同步清理关联的攻击结果与告警信息。通过该模块，用户能够快速完成从任务结果概览到攻击样本细节分析的完整过程。模块流程如图4.5所示。

### 4.4.5 历史记录管理模块设计

检索完成后，系统将符合条件的记录列表返回前端，并支持从历史记录页面快速跳转到检测详情页面进行进一步分析。该模块为长期检测结果追踪、实验归档和历史复盘提供了支持。模块流程如图4.6所示。

### 4.4.6 告警管理模块设计

告警管理模块主要用于管理检测过程中自动生成的异常流量告警信息。当检测结果中出现攻击流量时，系统会同步生成对应告警，并写入 `alarm_log` 表。管理员进入告警中心后，后端会查询告警内容、告警级别、处理状态及其关联检测记录信息，并返回前端进行展示。

在告警页面中，管理员可以根据实际处置情况更新告警状态，例如未处理、处理中和已处理等。状态更新后，系统会将修改结果持久化到数据库并返回最新告警列表。该模块将检测结果进一步转化为可用于安全运维的事件信息，增强了系统的实用价值。模块流程如图4.7所示。

### 4.4.7 模型管理模块设计

模型管理模块用于维护系统所接入的深度学习模型，是管理员进行算法管理和系统扩展的重要入口。管理员可以在该模块中新增模型记录，填写模型名称、模型路径、模型类型、评价指标、数据格式说明和所需字段列表等内容。系统在保存之前会校验模型路径和元数据是否有效，防止错误模型被加入检测链路。

若管理员选择启用当前新增模型，系统会自动取消其他模型的激活状态，并将该模型设置为当前唯一启用模型，从而保证检测任务始终基于明确的模型版本执行。通过该模块，系统实现了模型接入、模型切换和模型描述信息维护的统一管理。模块流程如图4.8所示。

### 4.4.8 性能评估模块设计

性能评估模块主要用于展示当前启用模型或指定模型的实验评估结果。管理员进入性能评估页面后，系统会优先检查本地是否存在缓存评估结果；若缓存存在，则直接读取 Accuracy、Precision、Recall、F1 值、混淆矩阵和测试样本数等信息；若缓存不存在，则重新执行测试集评估过程并生成新的评估结果文件。

前端根据返回数据将模型性能指标展示为卡片、表格和混淆矩阵区域，便于管理员从整体性能、分类效果和错误分布等多个角度分析模型表现。该模块将离线实验结果与在线管理平台打通，提高了系统对模型研究和应用分析的支撑能力。模块流程如图4.9所示。

### 4.4.9 流量监测模块设计

流量监测模块用于实现在线流量采集、特征提取和自动送检，是系统区别于普通离线检测平台的重要扩展功能。管理员首先配置抓包网卡、抓包时长和轮询间隔等参数，随后系统启动流量监测线程并周期性扫描网络流量。在线监测过程中，系统利用 Scapy 抓取原始流量并生成 PCAP 文件，再调用 Python cicflowmeter 将其转换为流量特征 CSV。

由于在线提取得到的列名与训练阶段的数据格式不完全一致，系统还会执行列名规范化过程，将其转换为模型可接受的输入结构。完成转换后，系统调用当前启用模型完成自动检测与告警生成，并将处理后的 PCAP/CSV 文件归档到指定目录。若监测线程继续运行，则系统进入下一轮抓包；若管理员停止监测，则流程结束。模块流程如图4.10所示。

用户管理模块用于维护系统中的账号信息和角色配置，仅对管理员开放。管理员可以在该模块中新增、编辑或删除用户，并填写用户名、密码和角色等信息。系统在写入前会校验用户名唯一性、角色合法性和密码规范性，避免出现重复账号或非法配置。

## 4.5 数据库设计

### 4.5.1 概念结构设计

结合系统功能需求与业务流程，数据库概念结构设计主要围绕系统用户、模型信息、数据集信息、检测任务、检测记录、攻击结果、告警日志和操作日志等核心实体展开。系统用户实体主要描述账号与角色权限信息；模型信息实体描述模型文件路径、启用状态及评价指标；数据集信息实体记录上传文件来源与样本规模；检测任务实体用于表示后台异步任务执行过程；检测记录实体用于保存每次检测任务的总体统计结果；攻击结果实体描述被识别出的异常流量；告警日志实体用于记录异常告警；操作日志实体则用于保存登录、模型切换、删除记录等关键操作信息。

从实体关系来看，一个系统用户可以对应多条检测任务和多条检测记录；一条检测任务在执行完成后可生成一条检测记录；一条检测记录可以对应多条攻击结果和多条告警日志；一个模型可以在逻辑上被多个检测任务和检测记录引用；操作日志与用户实体之间存在逻辑上的一对多关系。数据库概念结构设计图如图4.12所示。

### 4.5.2 逻辑结构设计

在逻辑结构设计阶段，本文采用关系型数据库对系统实体进行落地实现，并通过主键、外键和逻辑关联字段建立各表之间的联系。系统主要数据表及逻辑关系如表4.1所示。

表4.1 主要数据表逻辑关系说明

| 表名 | 主键 | 关键外键/关联字段 | 主要功能 |
| --- | --- | --- | --- |
| `user` | `id` | 无 | 保存系统用户账号、密码和角色信息 |
| `model_info` | `id` | 无 | 保存模型路径、格式说明、指标和启用状态 |
| `dataset_info` | `id` | 无 | 保存上传数据文件的来源与样本数 |
| `detection_task` | `id` | `user_id`、`record_id`、`model_id` | 保存后台检测任务状态与进度 |
| `detect_record` | `id` | `user_id`、`model_id` | 保存一次检测任务的统计结果 |
| `attack_result` | `id` | `record_id` | 保存攻击样本详细信息 |
| `alarm_log` | `id` | `record_id` | 保存异常告警内容与状态 |
| `operation_log` | `id` | `username`（逻辑关联） | 保存登录、切换模型、删除记录等操作行为 |

从映射关系上看，`user` 与 `detection_task`、`detect_record` 之间均为一对多关系；`detect_record` 与 `attack_result`、`alarm_log` 之间均为一对多关系；`model_info` 与 `detection_task`、`detect_record` 之间属于逻辑上的一对多引用关系；`detection_task` 与 `detect_record` 之间通过 `record_id` 建立任务执行结果映射。该逻辑结构能够较好地支撑系统中的异步任务调度、检测结果归档、模型切换和日志追踪等业务需求。

## 4.6 本章小结

本章主要对基于深度学习的入侵检测系统进行了总体设计。首先，从系统总体设计角度出发，说明了平台的设计目标与整体架构；其次，对数据预处理流程和 1D-CNN+BiLSTM+Attention 模型设计进行了分析；随后，按照用户登录与权限控制、系统首页与统计展示、数据上传与检测任务、检测结果展示、历史记录管理、告警管理、模型管理、性能评估、流量监测和用户管理十个功能模块，对系统功能结构和业务流程进行了详细设计；最后，从概念结构、逻辑结构和数据字典三个层面对数据库设计进行了说明。

通过本章设计，可以看出本文所构建的入侵检测系统已经形成了较完整的技术路线和功能框架，为后续系统实现与实验结果分析奠定了基础。
